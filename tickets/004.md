# Ticket 004 — Implement Baseline Drift Detection Tool

## Title

Create `check-drift.sh` to compare current artifacts against baseline before auditor execution

---

# Problem

The current SecurityCouncil pipeline relies on LLM auditors to determine whether the VM has drifted from baseline.

That works, but it is expensive and indirect for the most basic question:

> Did anything materially change?

Some artifact fields are deterministic and comparable without LLM help. Others are volatile and should be normalized or ignored.

There is currently no simple script that answers:

```text
baseline vs current: drift or no drift?
```

before the audit pipeline runs.

---

# Goal

Introduce a deterministic drift-checking script that compares:

```text
vm_artifacts/
vs
vm_baseline/artifacts.baseline/
```

and reports whether **material drift** exists.

This tool should:

* ignore expected volatile fields
* detect real system-state changes
* provide fast pre-audit feedback
* be usable independently of LLM auditors

---

# Scope

Create a new script:

```text
scripts/check-drift.sh
```

---

# Drift Model

The script must compare the following artifacts:

```text
config.json
packages.txt
ports.txt
firewall.txt
usage.json
```

Baseline location:

```text
vm_baseline/artifacts.baseline/
```

Current location:

```text
vm_artifacts/
```

---

# Volatile Fields To Ignore

The script must **ignore** the following fields when checking drift.

## JSON timestamps

Ignore:

```text
timestamp_utc
generated_at_utc
```

in any JSON files.

## Runtime process IDs and file descriptors in `ports.txt`

Ignore differences in:

```text
pid=<number>
fd=<number>
```

These should be normalized before comparison.

---

# Required Behavior

## Check 1 — Artifact presence

Verify both current and baseline copies exist for:

* config.json
* packages.txt
* ports.txt
* firewall.txt
* usage.json

If any pair is missing, exit with an error.

---

## Check 2 — Semantic comparison of `config.json`

Compare the two files after removing ignored timestamp fields.

Differences in actual config fields should be reported as drift.

Example drift:

* browser tool changes from false → true
* ollama access changes from false → true
* elasticsearch access changes

---

## Check 3 — Semantic comparison of `usage.json`

Compare after removing timestamp fields.

Differences in:

* anthropic_calls
* openai_calls
* gemini_calls
* cap_threshold_warning
* estimated_cost_usd

should be treated as drift.

---

## Check 4 — Semantic comparison of `ports.txt`

Normalize both files before comparison.

Required normalization:

* replace `pid=<number>` with `pid=PID`
* replace `fd=<number>` with `fd=FD`

Then compare normalized files.

Differences in:

* port numbers
* bound interfaces
* service names

should be treated as drift.

---

## Check 5 — Comparison of `firewall.txt`

Compare as plain text.

Differences in:

* default policies
* rules
* logging mode

should be treated as drift.

---

## Check 6 — Comparison of `packages.txt`

Compare package state deterministically.

The simplest acceptable version is a direct compare of the full file.

If later needed, package parsing may be added, but initial implementation may use:

```bash
diff
```

Differences should be treated as drift.

---

# Script Output

If no drift exists, print:

```text
No material drift detected
```

If drift exists, print:

```text
Material drift detected
```

and list which artifact(s) differ.

Example:

```text
Material drift detected:
- config.json
- ports.txt
```

---

# Exit Codes

The script must use:

* `0` → no material drift
* `1` → material drift detected
* `2` → validation / missing-file error

This makes the script usable in automation.

---

# Design Constraints

The script must:

* use `set -euo pipefail`
* be deterministic
* avoid LLMs entirely
* avoid modifying any files
* operate only on existing host-side artifact directories

---

# Acceptance Criteria

Running:

```bash
./scripts/check-drift.sh
```

should:

1. compare current vs baseline
2. ignore volatile fields
3. report drift/no-drift
4. exit with the correct code

Examples:

## Stable case

```text
No material drift detected
```

Exit code: `0`

## Drift case

```text
Material drift detected:
- firewall.txt
- ports.txt
```

Exit code: `1`

## Missing artifact case

```text
ERROR: Missing vm_artifacts/firewall.txt
```

Exit code: `2`

---

# Integration Target

This script should eventually run before LLM auditors:

```text
collector
 ↓
sync
 ↓
validate-artifacts.sh
 ↓
check-drift.sh
 ↓
auditors
 ↓
aggregator
```

This allows the system to distinguish:

* artifact validity
* deterministic drift
* LLM interpretation

---

# Out of Scope

This ticket does not:

* modify artifacts
* update baseline
* invoke LLMs
* generate reports

It only performs deterministic drift detection.

---

# Result

After implementation, the system gains a fast, reliable baseline comparison tool that reduces unnecessary auditor runs and gives immediate signal about material change.
